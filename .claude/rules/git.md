# Git & GitHub Workflow Rules

## 1. Branching Strategy
- **Naming Convention:** Use `type/short-description` (e.g., `feat/auth-flow`, `fix/docker-build`, `refactor/api-routes`).
- **Base Branches:** Always branch off the latest state of `main`. Run `git pull --rebase origin main` before creating a new branch to maintain a linear history.

## 2. Commit Standards
- **Conventional Commits:** Strictly use `feat|fix|refactor|docs|test|chore(scope): message`.
- **Atomic Commits:** Separate infrastructure/Docker changes from application code. Separate refactoring from new features. Do not bundle unrelated changes into a single commit.
- **Meaningful Bodies:** For `feat`, `fix`, and `refactor`, include a commit body explaining the *why* and the *impact* of the change.
- **CI/CD Awareness:** Append `[skip ci]` or `[ci skip]` when changes only affect Markdown, `.gitignore`, or static documentation. Note: with squash-merge the tag must be in the squash/merge commit message — that's the one CI reads, not the intermediate commits it absorbs.
- **NO AI ATTRIBUTION:** Never add a Claude co-author line or any AI attribution to commit messages (absolutely no `Co-Authored-By: Claude` or similar tags).

## 3. Tone & Writing Style
- Voice for commits, PRs, reviews, and comments is defined in `tone.md` — the single source. It is not restated here, so the two files can't drift apart.

## 4. Security & Staging
- **Pre-Commit Review:** Always run `git diff --staged` and review the output before generating a commit message. Never blind-commit.
- **Stage Deliberately:** Use `git add -p` or explicit paths so commits stay atomic (see §2). `git add .` stages everything at once and works against that separation — reserve it for a genuinely single-concern change.
- **Strict Secrets Policy:** Never stage or commit `.env` files, `.pem` keys, API tokens, or hardcoded credentials. Verify `.gitignore` covers them before staging.

## 5. Pull Requests, Reviews, and Issues
- **Merging:** Squash-merge feature branches into `main`.
- **History:** Never `git push --force` or `git push --force-with-lease` to `main`.
- **Identity (GitHub MCP):** Comments, reviews, and PR descriptions go out under the authenticated GitHub user — you are the one running and reviewing the change. No AI disclaimers, no "As an AI", no "Generated by Claude", no bot signatures. Write like a normal teammate (voice: `tone.md`). If someone directly asks whether a change was AI-assisted, answer honestly rather than deny it.
- **Link Tracking:** Automatically append `Closes #<issue-number>` or `Relates to #<issue-number>` to the PR description if an issue was mentioned in the prompt.
